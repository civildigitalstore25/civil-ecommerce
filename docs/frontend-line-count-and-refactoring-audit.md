## Code quality analysis (DRY, hardcoding, shared UI)

This section is a **manual / heuristic audit**, not an automated duplicate detector. Use it as a backlog for refactoring.

### Repetition and near-duplication

- **Product detail vs admin view**: FAQ patterns exist as `components/common/FAQItem.tsx` (marketing) and `pages/productDetail/ProductDetailFAQItem.tsx` (theme-aware). Consider a single FAQ row component with optional `colors` / `variant` props.
- **Header brand dropdowns**: `AutodeskDropdown` is already split into `Header/autodeskDropdown/*` (shell ~80 lines + submodules). **`AdobeDropdown`**, **`MicrosoftDropdown`**, and **`AntivirusDropdown`** remain ~238–242 lines each and likely share the same two-panel shell; a generic `BrandMegaMenu` driven by config (`HeaderConfig.ts` + data) would remove repetition.
- **Admin pagination**: Multiple `Pagination.tsx` files under `ui/admin/products`, `users`, `orders` — consolidate under `ui/admin/components/` if markup matches.
- **Feature / pricing cards**: Overlap between `components/common/FeatureCard.tsx`, `components/SCrm/sections/Features/*`, and `ui/ScrmSection/FeatureCard.tsx` — align naming and reuse one primitive where visuals match.
- **Reviews UI**: Storefront reviews (`productDetail/tabs/*`) vs admin `ui/admin/reviews/*` serve different roles but share concepts (stars, date format); extract tiny presentational pieces (e.g. star row, relative date) if duplicated strings/styles appear.

### Hardcoded values

- **Hex and fixed colors** still appear in many TSX files (e.g. `#2563eb`, `#ef4444`, `#fff`, `#10b981`) alongside `AdminThemeContext` usage. Prefer theme tokens (`colors.interactive.primary`, semantic names) or a small `theme/colorTokens.ts` for exceptions.
- **Magic numbers**: Spacing, animation durations, and z-index are often inline in large pages (`ProductDetail.tsx`, `CheckoutPage.tsx`). Prefer Tailwind tokens or shared constants where the same value repeats.
- **Copy / legal text**: Long strings in `ui/policy/*`, `ContactPage`, and auth pages may belong in `constants/siteContent.ts` (or CMS) if reused or translated later.

### Reusable UI and constants (already good / keep centralizing)

- Existing shared modules: `src/constants/` (`reviewConstants`, `productFormConstants`, `swalTheme`, `siteContent`), `src/utils/swal.ts`, `src/components/common/*`, `src/hooks/useAdminThemeStyles.ts`.
- **Continue** moving repeated Swal options, review filter options, and form field lists into `constants/` or colocated `*Constants.ts` next to the feature.
- **Extract** repeated inline button styles (primary/ghost/destructive) into shared button components or a thin `getAdminButtonStyle(variant)` helper tied to theme.

### What “done” would look like

- No file in `src/` routinely above **250 lines** without a documented reason (generated data files, single-purpose maps like `IconPicker`).
- Theme-aware surfaces use **theme context** (or CSS variables) instead of raw hex in JSX.
- **One pattern** per concern (FAQ row, mega menu, pagination, review star strip) imported from a single module.

## Remaining refactoring backlog (suggested order)

1. **`src/pages/ProductDetail.tsx`** — Still the largest file; extract modals, enquiry/edit flows, review-type modals, and any remaining layout sections into `productDetail/` (mirror tabs + purchase splits).

5. **`App.tsx`** — route table / lazy imports in `app/routes.tsx` (or similar).

7. **`ProductDetailReviewCard.tsx`** — split reply thread vs review header vs action bar.
8. **`utils/seo.ts`**, **`hooks/useCart.ts`**, **`contexts/CartContext.tsx`** — submodules by concern if splits stay coherent.
9. **CSS**: `WelcomePopup.css` — group by component section or co-locate with extracted TSX pieces.
